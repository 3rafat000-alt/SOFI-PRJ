تقرير التصميم المعماري: مركز ربط خدمات SAKK (Integrations Hub)

المرحلة: Gate 6 (Staging/UAT Consolidation)

المراجعة: مراجعة هندسية من المستوى الأول (Senior Architect Desk)

1. التقييم المعماري لنمط ServiceConfigOverrideProvider

تعديل الإعدادات في الـ Runtime وقت الإقلاع عبر الـ Service Provider هو النمط الأكثر عملية وتوافقية في بيئة Laravel 12 الخلفية لـ php-fpm وCloudflare لسبب جوهري: الحفاظ على الـ Open-Closed Principle. هذا النمط يسمح لك بتمرير أسرار البوابات ديناميكياً دون كسر كود الحزم الخارجية (3rd-party packages) التي تعتمد بشكل صارم ومباشر على استدعاء دالة المساعد config('services.x').

المفاضلات المعمارية (Trade-offs):

مزود التجاوز الحالي (Runtime Override): يمتاز بالتوافق التام وصفر تعديل على الخدمات المستهلكة. عيبه الأساسي هو استهلاك موارد المعالج والـ Database I/O عند كل طلب HTTP صادر عبر php-fpm لفك تشفير الأسرار وإعادة حقنها في الذاكرة.

حقن مستودع الإعدادات (Config Repository Injection): أكثر نقاءً برمجياً (Clean Code)، حيث تُحقن خدمة الإعدادات في الـ Constructor الخاص بالبوابة. عيبه: يتطلب إعادة هيكلة (Refactoring) واسعة النطاق لجميع الخدمات، وهو أمر مرفوض هندسياً في مرحلة متقدمة كـ Gate 6 لمنع الانحدارات صعبة التتبع.

إعادة التحميل الحدثية المُخبّأة (Cached Event-Driven): النمط الأمثل للأداء الفائق. يقرأ المزود الإعدادات من كاش سريع (مثل Redis) وقت الإقلاع؛ وعند قيام الأدمن بتحديث لوحة التحكم، يُطلق حدث يعيد بناء الكاش فوراً.

الحكم المعماري:

النمط الحالي صحيح ومقبول، بشرط تغليفه بطبقة كاش صلبة ومحمية تفصل الرنتايم عن قاعدة البيانات لتفادي اختناق الخادم (Database Connection Exhaustion) تحت طفرات الزوار القادمة من Cloudflare.

2. استراتيجية توحيد البيانات التكرارية (DRY Strategy) بلا هجرة خطرة

وجود الأسرار موزعة بين integrations وservice_configs والـ .env يمثل بيئة خصبة لحوادث "تشتت التكوين" (Configuration Drift). للتخلص من هذا التكرار بأمان ودون هجرة "الانفجار العظيم" (Big-Bang) الخطرة على محفظة حية، نتبع نمط الجسر المتدرج (Shadow Bridge Pattern):

هيكلية الحقيقة الموحدة (Single Source of Truth):

دمج الجدولين في جدول integrations فقط، مع تحويل حقل الاعتمادات إلى مصفوفة نصوص مشفرة عند الراحة (Encrypted Text Array).

[لوحة تحكم الأدمن] ───► تحديث الإعدادات ───► (Dual-Write) ───► 1. جدول integrations (المستقبل)
                                                                 2. جدول service_configs (مؤقت)
                                                                 
[مزود التجاوز Runtime] ───► قراءة التكوين ───► (Fallback Flow) ───► integrations ──► service_configs ──► .env

خطة العبور التدريجي (Zero-Downtime Migration):

المرحلة أ (الكتابة المزدوجة - Dual Write): تعديل الـ Controller في لوحة التحكم ليقوم بحفظ البيانات مشفرة في الجدول الجديد integrations وفي نفس الوقت يكتبها في الجدول القديم service_configs.

المرحلة ب (القراءة المتراتبة - Layered Fallback): تعديل الـ ServiceConfigOverrideProvider ليقرأ من جدول integrations أولاً؛ فإذا كان السجل مفقوداً أو فارغاً، يرتد صامتاً لـ service_configs ثم إلى الـ .env. هذا يحمي الـ OTP والخدمات الحية من أي انقطاع.

المرحلة ج (التطهير - Deprecation): بعد التأكد من استقرار بيئة Staging، يتم تشغيل سكربت خلفي (Migration Script) ينقل ما تبقى من السجلات القديمة، ثم تُحذف الجداول الزائدة والأسطر الحساسة من ملف .env الإنتاجي نهائياً.

3. هندسة مفتاح القتل لـ SEV-4 (Fail-Open vs Fail-Closed)

يكمن التحدي المعماري هنا في التوفيق بين أمرين متناقضين: عدم إسقاط التطبيق بالكامل إذا أخطأ الأدمن في ضبط صف (Fail-Open)، وفي نفس الوقت الإيقاف الصارم للبوابة المالية إذا تم تفعيل الـ Toggle OFF لمنع الارتداد للـ env (Fail-Closed).

التصميم البرمجي للحماية المزدوجة:

نصنف الخدمات داخل الـ Provider إلى فئتين: خدمات مالية (حاسمة) وخدمات تشغيلية (مساندة).

PHP
public function boot(): void
{
    // 1. تحديد الفئات الصارمة
    $financialGateways = ['ccpayment', 'stripe', 'stripe_issuing'];
    
    try {
        $configs = Integration::where('is_active', false)->get()->keyBy('key');
        
        foreach ($financialGateways as $gateway) {
            // إذا وجد صف معطل يدوياً من الأدمن -> سحق وتصفير الإعدادات فوراً (Fail-Closed)
            if ($configs->has($gateway)) {
                config(["services.{$gateway}" => [
                    'app_id' => null, 'secret' => null, 'is_active' => false
                ]]);
                continue;
            }
            
            // في حال وجود صف فعال، يتم شحن الاعتمادات بشكل طبيعي
            if ($activeRow = Integration::where('key', $gateway)->where('is_active', true)->first()) {
                config(["services.{$gateway}" => $activeRow->credentials]);
            }
        }
    } catch (\Throwable $e) {
        // الـ Catch هنا يحمي التطبيق من الانهيار الكلي إذا سقطت الـ DB (Fail-Open للتطبيق)
        // لكنه يضمن تصفير بوابات الدفع احتياطياً كـ Fail-Closed مالي
        foreach ($financialGateways as $gateway) {
            config(["services.{$gateway}.is_active" => false]);
        }
        logger()->critical('Integrations Hub Failure: Safe financial lockdown triggered.', ['error' => $e->getMessage()]);
    }
}


الأثر الهندسي: لو وضع الأدمن بوابة CCPayment على OFF، يتم مسح قيم الـ Config في الذاكرة تماماً، مما يمنع الـ .env Fallback من إحياء البوابة الميتة، وفي حال تعطل قاعدة البيانات يقع النظام في المنطقة الآمنة (شاشات الأدمن تعمل، بوابات الأموال مقفلة لحين الإصلاح).

4. ترتيب المرحلة-2 وتدقيق مخاطر السطح Tier-A
الترتيب التنفيذي الموصى به:

إصلاح SEV-4 (مفتاح القتل الصارم): الأولوية القصوى لحماية النواة المالية من الارتدادات غير المحكومة.

إصلاح SEV-5 (تروائس الـ API ومصائد الـ 302): لضمان وضوح عقود البيانات وعرض الأخطاء الحقيقية 422 للأدمن والموبايل بدلاً من التوجيه الأعمى.

إصلاح SEV-7 (سجل التدقيق للأسرار): أمن حوكمة داخلي حرج لمنع الاحتيال الداخلي (Internal Fraud).

إصلاح SEV-6 (زر فحص الاتصال الحقيقي): أداة تشخيصية ممتازة تُترك لنهاية السبرت.

الرد على الملخص الإيجازي المطلق

نعم، خطة ترتيب المرحلة-2 صحيحة تماماً هندسياً، لأن معالجة الثغرات التشغيلية وبوابات التحكم بالأموال (SEV-4/5) يجب دائماً أن تسبق بناء أدوات التشخيص والتدقيق اللاحقة (SEV-6/7).

3 مخاطر جوهرية تم تقليل وزنها وأُغفلت في التقرير:

إنهاك المعالج (Decryption CPU Exhaustion): فك تشفير الأسرار الحساسة لكل الخدمات عبر دالة Crypt::decryptString عند كل طلب HTTP صادر تحت مظلة php-fpm سيتسبب في قفزة حادة باستهلاك المعالج (CPU Spikes) وارتفاع زمن الاستجابة (Latency) تحت ضغط Cloudflare الحقيقي؛ يجب إجبار المزود على استخدام كاش مشفر (Encrypted Tagged Cache).

كارثة الـ الارتداد الصامت لبيئة التطوير (Silent Dev-Fallback): في حال حدوث سقوط لحظي لاتصال قاعدة البيانات، فإن منطق الـ Fail-Open الحالي سيرتد صامتاً لقراءة الـ .env. إذا كان الـ .env يحتوي على مفتاح Stripe Test أو حساب Sandbox قديم، ستتحول حركات المستخدمين الحقيقية لبيئات وهمية صامتة دون إطلاق إنذار؛ يجب تصفير قيم الـ env تماماً للبوابات الحيوية في الإنتاج.

حقن التوكنات في الـ Error Logs (Leaking via Context): عند إصلاح SEV-5 وجعل نداءات الـ Fetch تعيد الأخطاء كاملة، هناك خطر حاد قيام الـ Global Exception Handler أو الـ Loggers بابتلاع الـ Request Payload عند حدوث خطأ 422 أو 500، مما يتسبب في كتابة أسرار ومفاتيح الـ APIs الحية داخل ملفات الـ Laravel Logs بنص صريح (Plain Text)؛ يجب إعداد مصفوفة الـ dontFlash و hidden في سجلات النظام بدقة صارمة.
