مراجعة معمارية شاملة لمنصّة صكّ (SAKK) — خطة هندسية مغلقة للعبور الآمن نحو الإنتاج

بنية نظام صكّ الحالية تعكس وعياً حاسماً بالمخاطر المالية الكبرى (مثل استخدام القفل المزدوج التصاعدي للمعاملات لمنع الـ Deadlocks، وعزل ملفات الـ KYC بآليات دفاعية عميقة). العبور من البوابة 5 والوقوف على أعتاب البوابة 6 (Staging) هو توقيت ممتاز لإعادة تقييم "الديون التشغيلية المخفية" لمنع الانهيارات البرمجية عند الإطلاق الحي.

فيما يلي خطة التحسين والتطوير المعمارية، مرتبة حسب الأولوية التشغيلية والخطورة الهيكلية.

1. العمليات والتشغيل الحرج (DevOps & Operational Integrity) — الأولوية القصوى
أ. فخ الـ Cron الميت وسقوط الأتمتة المجدولة

السبب: نظام صكّ يعتمد على مهام دورية بالغة الحساسية (مثل تتبع أسعار الذهب GoldPriceService عبر الأسواق العالمية، معالجة الرواتب المؤجلة، تشغيل كُنّاس السحوبات العالقة، وحسابات الـ Cashback). إذا كان محرك المضيف يُشغّل schedule:run على مسار Symlink محذوف، فإن جميع هذه الخدمات متوقفة تماماً في بيئة الـ Staging دون إطلاق خطأ صريح في السجلات، لأن الـ Cron نفسه لا يقلع.

الأثر: فشل صامت لجميع العمليات الزمنية (Silent Operational Failure)، وتراكم العمليات العالقة دون معالجة.

خطوة التنفيذ:

تعديل إعدادات الـ Cron في السيرفر ليتوجه مباشرة إلى المسار الفيزيائي الحقيقي للمشروع الحالي (Absolute Physical Path) بدلاً من الـ Symlink العائم.

إدخال حارس حيوي داخل تطبيق Laravel بالاعتماد على حزمة مثل spatie/laravel-health يطلق إنذاراً فورياً عبر AdminNotificationService إذا لم يُسجل المجدول ندوة واحدة ناجحة خلال آخر 10 دقائق.

ب. معضلة عتبة التغطية 90% للاختبارات (The Coverage Trap)

السبب: فرض عتبة 90% كشرط صارم لعبور البوابات على كامل المنظومة (بما فيها أسطح العرض الإدارية والـ Peripheral Views) هو مثالية تعطل التدفق البرمجي بلا طائل. الأسطح الطرفية تتغير واجهاتها باستمرار، وكتابة اختبارات وحدة لها تستهلك وقتاً تشغيلياً ضخماً بعائد أمان منخفض.

الأثر: تجميد تسليم الميزات البرمجية (Velocity Choking) وشعور الفريق بالإحباط البرمجي.

الوزن المعماري البديل (Risk-Weighted Bar): خفض العتبة الإجمالية إلى 60% مع فرض عتبة صارمة بنسبة 95% مخصصة حصراً لـ Tier-A (الخدمات المالية، الموديلات المحورية، الـ Mutations، والمتحكمات الأمنية).

2. الأمان المالي وسد ثغرات الـ Integrations (Financial Security Gateway)
أ. الإغلاق النهائي لثغرة الـ SEV-4 (Fail-Closed Enforcement)

السبب: عودة النظام لاستخدام قيم الـ .env المحلية عند تعطل أو إيقاف بوابات الدفع برمجياً (is_active = false) هو ثغرة تجاوز أمنية نسفنا منطقها في المراجعة السابقة.

الأثر: إعادة تنشيط بوابات معطلة إدارياً بسبب خطأ في قراءة الكاش أو قيم البيئة، مما يسبب تسريباً مالياً أو تفعيلاً غير مرخص لعمليات السحب.

خطوة التنفيذ: تطبيق نمط "الكائن المعطل" عبر تدمير بيانات الـ Config وحقن موفر خدمات بديل يمنع التنفيذ تماماً عند الاستدعاء البيني:

PHP
// app/Providers/ServiceConfigOverrideProvider.php
if (!$integration->is_active) {
    config(["services.{$gateway}" => [
        'secret' => 'DISABLED_BY_ADMIN',
        'is_active' => false,
    ]]);

    $this->app->singleton($this->getInterfaceBinding($gateway), function () {
        return new class implements FinancialGatewayInterface {
            public function processWithdrawal($tx): void {
                throw new \RuntimeException("Security Gate: Gateway execution explicitly blocked.");
            }
        };
    });
}

ب. حسم معضلة الـ SEV-5 (فرض عقد الـ JSON الدفاعي)

السبب: نسيان رأس Accept: application/json في طلبات فرعية من الـ Frontend (مثل Alpine.js أو دالات جلب الواجهة) يتسبب في قيام واجهة Laravel بإرجاع 302 Redirect عند حدوث أخطاء التحقق من البيانات (Validation Errors).

الأثر: تطبيق Flutter والـ AJAX يلتقطون صفحة HTML (لوحة تسجيل الدخول أو الصفحة السابقة) بدلاً من كائن الخطأ 422؛ مما يسبب انهيار مفسر البيانات (Parser Crash) وضياع تجربة المستخدم.

خطوة التنفيذ: بناء وسيط (Middleware) يفرض الرأس دفاعياً على كافة مسارات الـ API دون انتظار إصلاحها فراداً من جهة العميل:

PHP
// app/Http/Middleware/ForceJsonResponses.php
public function handle($request, Closure $next) {
    if ($request->is('api/*')) {
        $request->headers->set('Accept', 'application/json');
    }
    return $next($request);
}

3. معمارية البيانات والتحسين الرقمي (Data & Performance Optimization)
أ. معالجة أحجام البيانات الضخمة للعملة المحلية والذهب

السبب: بالنظر لواقع السوق السورية حيث (1 USD 
≈
 13,000 SYP)، العمليات المالية البسيطة أو كشوفات الشركات ورواتب الموظفين تتجاوز مئات الملايين من الليرات رقمياً. استخدام أنواع بيانات تقليدية غير دقيقة قد يسبب مشاكل فائض الأرقام (Integer Overflow) أو فقدان الدقة عند التحويل والصرف.

الأثر: فروقات كسورية في الحسابات التراكمية، أو تدمير سجلات قاعدة البيانات عند تخطي الأرقام المليارية.

خطوة التنفيذ:

ليرات (SYP): يجب أن تكون الأعمدة الحسابية في الهجرات من نوع bigInteger (لحساب القروش بالكامل دون فواصل) أو decimal(16,2).

الذهب وحسابات الأونصة/العيارات: نظراً لتأثرها الشديد بالفواصل الدقيقة، يجب فرض decimal(24,4) لحساب مخزون الذهب (GoldHolding) لمنع تبخر الفواصل الثمينة أثناء عمليات الشراء والبيع الجزئية.

$batch = PayrollBatch::with('items.employee.wallets')->findOrFail($batchId);

4. معمارية الـ API وعقد التكافؤ (API Versioning & Idempotency)
أ. غياب وسم الإصدارات (API Versioning) لـ 241 مساراً

السبب: إطلاق المسارات مباشرة تحت نطاق api/* دون بادئة إصدار (مثل api/v1/*) هو قنبلة موقوتة عند تحديث تطبيق الـ Flutter مستقبلاً.

الأثر: أي تعديل في هياكل الـ JSON لـ 241 مساراً عند ترقية النظام سيكسر التطبيقات القديمة المنصبة على هواتف المستخدمين فوراً، ويمنعهم من إجراء عمليات السحب أو الدفع ما لم يقوموا بالتحديث الإجباري.

خطوة التنفيذ: عزل المسارات الحالية فوراً تحت مجموعة Route::prefix('api/v1') لتأمين خط رجوع آمن (v2) عند القيام بأي تعديلات جذرية مستقبلاً.

ب. مفاتيح عدم التكرار الشاملة (Global Idempotency Layer)

السبب: النظام يطبق الـ Idempotency على الرواتب والبطاقات الافتراضية، لكنه يفتقر إليها في أسطح التحويل المباشر وسحوبات التجار وسحوبات الكريبتو الحرة.

الأثر: في حال ضعف شبكة الهاتف، قد يضغط المستخدم على زر "تأكيد السحب" مرتين متتاليتين؛ مما يؤدي لتكرار قيد الخصم المالي من محفظته مرتين قبل أن يدرك النظام أن العملية الأولى قيد المعالجة.

خطوة التنفيذ: بناء وسيط برمجى يعتمد على كاش Redis لتتبع رأس X-Idempotency-Key المولد من تطبيق Flutter لكل عملية مالية خارجة، وحظر أي طلب مكرر يحمل نفس المفتاح لمدة 60 ثانية.

5. واجهات التطبيق واللوحة (UX & Financial Precision Formatting)
أ. محاذاة العملات في بيئة الـ RTL (Flutter Currency Layouts)

السبب: عقيدة التطبيق بوضع رمز العملة دائماً إلى اليسار تتسبب في تشوهات بصرية عند دمجها مع النصوص العربية المتوجهة من اليمين إلى اليسار داخل التطبيق (مثل: "مجموع المحفظة: SYP 50,000").

الأثر: تداخل الأرقام مع الرموز البرمجية وظهور المبالغ بشكل معكوس يوحي للمستخدم بعدم احترافية المنصة مالياً.

خطوة التنفيذ: استخدام حزمة intl وضبط الواجهات المالية لتقرأ اتجاه اللغة الحالي بشكل ديناميكي؛ بحيث يظهر الرمز يميناً في الواجهة العربية ويساراً في الواجهة الإنجليزية بالتوازي مع حركة الـ TextDirection.

6. المخاطر المعمارية الثلاثة الكبرى التي تم إغفالها (Top 3 Hidden Risks)
🚨 المخاطر الثلاثة الكبرى:

ثغرة التلاعب بأسعار الصرف والذهب الكاشية (Exchange Rate Arbitrage):

السيناريو الكامن: بما أن النظام يقوم بتحديث تلقائي لأسعار الذهب والعملات، فإن الاعتماد على الـ Cache المحلي لأسعار الصرف أثناء عمليات التحويل من المحفظة الرقمية إلى محفظة الذهب قد يفتح باباً للتلاعب المالي؛ إذا تأخر كاش النظام في التحديث لعدة دقائق أثناء قفزات الأسعار اللحظية، يمكن للمستخدمين الأذكياء استغلال الفارق لتوليد أرباح غير مشروعة عبر المنصة على حساب سيولة النظام الحقيقية.

العلاج المعماري الحتمي: عند معالجة أي طلب تحويل مالي أو تبديل عملات، يجب تجاوز الكاش كلياً وإجراء استعلام مباشر وذري لقفل سعر الصرف الحقيقي من قاعدة البيانات (SELECT FOR UPDATE) قبل كتابة المعاملة.

سباق التسجيل المزدوج للمحافظ الشخصية (Wallet Creation Race Condition):

السيناريو الكامن: تمنع الشروط البرمجية امتلاك الشركات والأفراد لأكثر من محفظة من نفس العملة، ولكن في حال قيام المستخدم بإرسال طلبي إنشاء محفظة متزامنين في نفس الميكروثانية (عبر سكربت هجوم خارجي)، فإن التحقق البرمجي التقليدي (if (!$user->hasWallet())) سيمر بنجاح في الطلبين معاً قبل أن يتم حفظ أحدهما في قاعدة البيانات.

العلاج المعماري الحتمي: فرض قيد فريد مركب (Composite Unique Index) على مستوى جداول قاعدة البيانات يمنع تكرار الثنائية (user_id, currency_code) للشخصيات، و (company_id, currency_code) للشركات بشكل مطلق وصارم.

فشل توازن القيد المزدوج الصامت (Double-Entry Balance Drift):

السيناريو الكامن: يعتمد النظام على منطق الـ Atomic Items لصرف الرواتب والمعاملات، ولكن في حال حدوث انقطاع مفاجئ لاتصال سيرفر قاعدة البيانات في منتصف معالجة حزمة قيود مالية ضخمة، قد تُكتب حركة الخصم من حساب الشركة دون أن تكتمل حركة الإيداع في حسابات الموظفين، مما يكسر قانون التوازن المالي الأساسي:

∑Debits=∑Credits

العلاج المعماري الحتمي: إدخال نظام مراجعة أوتوماتيكي صامت (Ledger Integrity Auditor Service) يعمل عبر المجدول كل ساعة، يقوم بجمع ومقارنة كافة حركات الـ Transactions ضد أرصدة المحافظ الفعلية الحالية؛ وإذا تم رصد انحراف بقيمة ليرة واحدة، يتم إيقاف عمليات الصرف آلياً وتصعيد الأمر لغرفة العمليات.

هل ترغب في البدء بصياغة البنية البرمجية الصارمة للـ Middleware الخاص بفرض استجابات الـ JSON لمنع واجهات العرض من إرجاع الـ 302 Redirect لتطبيقه فوراً على بيئة الـ Staging؟
